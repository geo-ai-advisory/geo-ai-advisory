# Audio → AI Call Bot Script

## Overview

Расшифровывает аудиозаписи звонков (колл-центр, менеджер-клиент) через локальный Whisper и на основе расшифровок создаёт готовый скрипт для AI колл-бота с ветками возражений.

**Команда:** `/audio_script [папка с записями]`

**Что создаёт:**
- `TRANSCRIPT_RAW.md` — расшифровки каждого звонка с разметкой «Менеджер / Клиент»
- `CALLBOT_SCRIPT.md` — готовый скрипт для AI-бота: системный промпт, ветки, стоп-фразы, JSON-конфиг

**Требования:**
- Python 3 + `mlx-whisper` + `ffmpeg`
- Claude Code с доступом к Bash

---

## КРИТИЧНО: только mlx_whisper (GPU)

**ЗАПРЕЩЕНО** использовать `whisper` (openai-whisper) — он работает на CPU и занимает 30-80 минут.
**ОБЯЗАТЕЛЬНО** использовать `mlx_whisper` — работает на Apple Silicon GPU (Metal), в 5-10x быстрее.

### Установка (один раз)

```bash
pip3 install --break-system-packages mlx-whisper
brew install ffmpeg
```

### Проверка

```bash
which mlx_whisper  # должен быть /opt/homebrew/bin/mlx_whisper
```

### Модели

Рекомендуется `mlx-community/whisper-small-mlx` для русского языка.
Модель скачивается автоматически при первом запуске.

## Пошаговый сценарий

### 1. Найти аудиофайлы

```bash
# Найти все аудио в папке (mp3, m4a, wav, ogg)
find "[ПАПКА]" -type f \( -name "*.mp3" -o -name "*.m4a" -o -name "*.wav" -o -name "*.ogg" \)
```

Если папка не указана — спросить пользователя.

### 2. Расшифровать через mlx_whisper (Apple Silicon GPU)

**ВАЖНО:** Всегда использовать `mlx_whisper` вместо `openai-whisper` — работает на GPU (Metal), в 5-10x быстрее и не грузит CPU.

```bash
mlx_whisper file.webm --model mlx-community/whisper-small-mlx --language ru --output-dir /tmp/whisper_out --output-format txt
```

**Для длинных записей (>30 мин):** использовать large-v3 + anti-hallucination флаг:
```bash
mlx_whisper file.mp3 --model mlx-community/whisper-large-v3-mlx --language ru --condition-on-previous-text False --output-dir /tmp/whisper_out --output-format txt
```
Small и medium галлюцинируют на тишине/паузах в длинных записях (повторяют одну фразу сотни раз). `--condition-on-previous-text False` отключает петли.

Для нескольких файлов запускать параллельно через `run_in_background`.

**Установка (один раз):** `pip3 install --break-system-packages mlx-whisper`
**Если ffmpeg не найден:** установить по инструкции выше, затем перезапустить.

### 3. Разметить расшифровки

Сырой текст Whisper — сплошной, без разделения спикеров.
Нужно вручную (на основе контекста) разбить на реплики «Менеджер» / «Клиент»:

- Первая реплика — всегда менеджер (он инициатор звонка)
- Менеджер говорит структурно, представляется, объясняет причину звонка
- Клиент отвечает коротко, часто с вопросами или возражениями
- Переключение определяется по смысловым маркерам: «Здравствуйте», «Скажите», «Понял», вопросительные обороты

Записывать в `TRANSCRIPT_RAW.md`:

```markdown
## Звонок N — HH:MM | +7 XXX XXX-XX-XX

**Менеджер:** [реплика]

**Клиент:** [реплика]

**Менеджер:** [реплика]
...
```

### 4. Извлечь паттерны

Проанализировать все расшифровки и выделить:

1. **Стандартное открытие** — как менеджер представляется и называет причину звонка
2. **Типовые возражения клиентов** — все уникальные причины незавершённой сделки
3. **Успешные ответы** — формулировки, которые менеджер использовал для снятия возражений
4. **Ключевые USP** — аргументы, которые менеджер называл (цена, удобство, кэшбэк, сохранённые данные, онлайн-урегулирование и т.д.)
5. **Финал** — как заканчиваются успешные и нейтральные звонки

### 5. Написать CALLBOT_SCRIPT.md

Структура файла:

```markdown
# Скрипт AI колл-бота — [Тема]

## СИСТЕМНЫЙ ПРОМПТ (для AI-бота)
[Описание роли, задачи, тональности, переменных {CLIENT_NAME} и др.]

## СЦЕНАРИЙ РАЗГОВОРА

### 1. ПРИВЕТСТВИЕ
[Дословная фраза открытия]

### 2. ВЕТКИ ОТВЕТОВ

#### 2.1 Возражение: "[типовое возражение из записей]"
**Ответ:** [формулировка, основанная на реальных ответах менеджера]
→ [условие перехода к следующей ветке]

#### 2.2 Возражение: "..."
...

#### 2.N ЗАВЕРШЕНИЕ РАЗГОВОРА
[Финальная фраза]

## КЛЮЧЕВЫЕ ФАКТЫ (для встраивания в промпт)
[Таблица: параметр / значение]

## СТОП-ФРАЗЫ
[Список триггеров завершения + реакция бота]

## ПАРАМЕТРЫ ЗВОНКА (технические)
[JSON-конфиг: таймаут, перезвон, эскалация на оператора]
```

---

## Правила качества скрипта

- **Не придумывать** аргументы — только то, что реально использовал менеджер
- **Дословные формулировки** из расшифровок — не перефразировать без необходимости
- **Каждое возражение** из записей должно стать отдельной веткой
- **Тональность**: ненавязчиво, без давления — если клиент говорит «подумаю», принять и красиво завершить
- **Переменные** в системном промпте: `{CLIENT_NAME}`, `{PRODUCT_NAME}`, `{EXPIRY_DATE}` и т.д. — под конкретную интеграцию
- **JSON-конфиг** — добавить `transfer_to_human_on` для сложных сценариев (претензии, ДТП, юридические вопросы)

## Частые проблемы

| Проблема | Решение |
|----------|---------|
| `FileNotFoundError: ffmpeg` | `brew install ffmpeg` |
| `pip: externally-managed` | Добавить флаг `--break-system-packages` |
| Плохое качество расшифровки | Сменить модель на `mlx-community/whisper-medium-mlx` |
| Имена перепутаны (Whisper галлюцинирует) | Это нормально для имён собственных — исправить вручную при разметке спикеров |
| Звонок на двух языках | Убрать `--language ru` — Whisper определит автоматически |
| Используется `whisper` вместо `mlx_whisper` | **СТОП.** Команда `whisper` = CPU = 30-80 минут. Использовать ТОЛЬКО `mlx_whisper` |
| Транскрипт оборвался / модель галлюцинирует | **НЕ ПРОДОЛЖАТЬ** с частичными данными. Эскалация ниже ↓ |

---

## КРИТИЧНО: неполная транскрипция

Если транскрипт оборвался, содержит повторы или галлюцинации — это ПРОВАЛ, не "достаточно".

**Запрещено:** решить что "хватит" и перейти к построению отчёта / скрипта на неполных данных.

**Обязательно:**
1. Попробовать модель побольше: `mlx-community/whisper-medium-mlx` или `mlx-community/whisper-large-v3-mlx`
2. Разбить аудио на чанки по 10-15 минут через `ffmpeg -ss ... -t ...` и транскрибировать по частям
3. Попробовать без `--language ru` (автодетект)
4. Если всё провалилось → **СТОП** → доложить пользователю:
   - Что получилось, что нет (конкретные минуты / проценты)
   - Какие подходы попробовал
   - Спросить: «Продолжить с тем что есть или попробовать X?»
5. **Без ответа пользователя — НЕ ПРОДОЛЖАТЬ.**

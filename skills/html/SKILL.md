# HTML - генерация HTML-отчёта

## Вызов
`/html [задача]`

Примеры:
- `/html отчёт по dashboard и витрине`
- `/html сделать html-презентацию по проекту`
- `/html собрать standalone html-документ`

## Задача skill

Этот skill отвечает **только за генерацию HTML-артефакта**.

Он НЕ публикует файл в GitHub Pages.
Для публикации готового HTML использовать отдельный skill `/html-push`.

## Поведение

### 1. Сначала понять тип HTML
Определить, что именно нужно пользователю:
- отчёт
- презентация
- roadmap / architecture doc
- standalone landing / dashboard-like page
- визуальный документ по любой теме

### 2. Сначала взять референсы и стиль пользователя
Обязательно:
1. Найти минимум один сильный HTML-референс в текущем проекте.
2. Свериться с утверждённой дизайн-памятью проекта.
3. Если пользователь прислал скрины/референсы - они приоритетнее дефолтного стиля.
4. Не привязываться к одной жёсткой структуре отчёта: темы и содержание могут быть разными.

### 3. Правило skill: жёсткий процесс, гибкий контент
`/html` должен быть строгим по процессу, но не по шаблону содержания.

**Жёстко обязательно:**
- взять референсы
- собрать локальный `.html`
- сделать визуально сильный документ, а не text dump
- дать `file://` ссылку
- при возможности проверить layout в браузере до публикации

**Не должно быть жёстко зафиксировано:**
- единая структура секций для всех тем
- один и тот же состав блоков для всех документов
- один и тот же тип отчёта для любых задач

### 4. Дизайн-правила `/html`
`/html` должен соблюдать именно **дизайн-систему**, а не фиксированный сценарий контента.

Обязательные дизайн-правила:
1. Светлая тема.
2. Inter через `@import`.
3. Градиентный hero / сильная шапка.
4. Воздушность: достаточно внешних и внутренних отступов.
5. Белые карточки с тенью.
6. Тёмные акцентные блоки допустимы, но без гигантских пустых масс и с крупным текстом.
7. Технические блоки (API, endpoints, paths, code-ish данные) должны визуально отличаться:
   - `code` / pill / моноширинный стиль
   - аккуратные акцентные рамки
   - более явная иерархия
8. Визуальная иерархия должна быть сильной:
   - hero заметно крупнее остального
   - меню читаемое, не микроскопическое
   - KPI и карточки не должны быть слишком мелкими
9. Общая ширина и scale можно адаптировать под задачу. Не держать жёстко один `max-width`, если из-за этого всё выглядит слипшимся или слишком мелким.
10. Документ должен выглядеть как презентация/visual-doc, если пользователь ждёт именно этого.
11. Любой длинный текстовый блок (больше одного абзаца или списка фактов) НЕЛЬЗЯ подавать сплошным полотном. Нужно раскладывать его в подходящую структуру: карточки, таймлайны, схемы, таблицы, сравнения, KPI-блоки, диаграммоподобные секции.
12. Формат структуры подбирать по смыслу:
   - хронология → timeline
   - сравнение / плюсы-минусы → 2-колоночные карточки
   - процесс → шаги / flow
   - числовая динамика → KPI / mini-chart / progress
   - сложный narrative → разбивка на короткие карточки с подзаголовками
13. Если визуально можно сделать понятнее через схему, карточки или таймлайн — нужно делать именно так, а не оставлять длинний текст.
14. У всех кнопок и кликабельных controls ОБЯЗАТЕЛЬНО должны быть минимум 3 визуально различимых состояния: default / hover / active (pressed). По кнопке должно быть сразу понятно, что это кнопка.
15. Hover должен быть заметным: изменение фона, border, тени, текста или лёгкий lift. Active/pressed тоже должен отличаться от hover и default.
16. Недопустимы «мертвые» кнопки без реакции на наведение и нажатие — это воспринимается как сломанный UI и нарушает ощущение кликабельности.
17. Если в документе есть меню, табы, nav-pills, CTA или secondary actions — это тоже кнопки/controls и на них распространяется правило 3 состояний.
18. Перед отдачей HTML визуально проверить именно интерактивность: можно ли без подсказки понять, что элемент кликабелен, есть ли hover, есть ли pressed.
19. Для кнопок по умолчанию использовать transition, pointer cursor, hover-state и active-state, а не только static pills.
20. Если пользователь один раз указал на отсутствие hover/active у кнопок, считать это жёстким правилом для всех следующих HTML-задач.

### 5. Проверка до отдачи
Перед тем как считать результат готовым:
1. Сохранить локальный `.html` файл.
2. Сразу дать пользователю `file://` ссылку.
3. Если доступна browser-check - открыть локально и проверить:
   - scale шрифтов
   - отступы
   - геометрию карточек/бейджей/кружков
   - отсутствие слепленности
   - баланс тёмных и светлых блоков
4. Исправить визуальные проблемы до публикации.

## Правила
- Не публиковать HTML через этот skill.
- Не делать ручной password-wrapper.
- Не смешивать генерацию HTML и публикацию в один шаг.
- Если пользователь просит HTML-артефакт - результатом должен быть локальный `.html` файл + `file://` ссылка.
- Skill задаёт **дизайн-правила и процесс**, а не одну жёсткую структуру отчёта на все случаи.

## После выполнения
- Сохранить HTML в подходящее место проекта.
- Сразу прислать `file://` ссылку.
- Если нужна публикация - использовать `/html-push <path-to-html> [repo-name] [password:<пароль>]`.
- Если пользователь одобрил конкретный визуальный стиль - придерживаться именно его в следующих HTML-задачах, пока не скажет иначе.

## Что важно помнить
- HTML-отчёты в этом проекте могут быть на совершенно разные темы.
- Одинаковыми должны быть не разделы, а качество подачи: масштаб, воздух, читаемость, акценты, карточки, hero, typography.
- Если есть конфликт между старым дефолтным шаблоном и присланным пользователем референсом, референс пользователя важнее.

## Примеры использования
- `/html отчёт по dashboard и витрине`
- `/html архитектурный документ по API`
- `/html презентация как работает продуктовая команда`
- `/html визуальный документ по новой фиче`
- `/html standalone html-документ по любой теме без публикации`

## Публикация отдельно
Для публикации готового HTML использовать только:
`/html-push <path-to-html> [repo-name] [password:<пароль>]`

`/html-push` отвечает только за GitHub Pages, пароль и ссылку.
`/html` отвечает за качество самого HTML.

## HTML-дизайн система - кратко
Базовые принципы:
- hero должен быть сильным и заметным
- карточки должны быть крупными и читабельными
- текст не должен выглядеть микроскопическим
- технические данные должны визуально отличаться от narrative-блоков
- document scale подгоняется под восприятие, а не под один захардкоженный размер
- итоговый HTML должен выглядеть «максимально сильно», а не как черновой документ

Строгая структура секций НЕ обязательна. Важны визуальные правила и качество подачи.

## Связанные правила проекта
- См. feedback-память про HTML-дизайн и layout
- См. project references в папке документации вашего проекта (обычно `docs/` внутри проекта)
- Скрины и референсы пользователя приоритетнее дефолтного шаблона
- Сначала сделать сильный локальный HTML, потом уже публиковать через `/html-push`

## Идеальный mental model
`/html` = дизайнер+верстальщик HTML-артефакта
`/html-push` = курьер, который только публикует готовый файл

Не путать эти роли.

# Глобальные правила исполнительной дисциплины

## 0. Журнал задачи и извлечение навыков

### Когда создавать журнал

**По умолчанию — СОЗДАВАЙ.** Журнал нужен для любой сессии где ты делаешь что-то осмысленное. Если сомневаешься — создавай.

Создавай журнал когда:
- Выполняешь любую задачу с результатом (отчёт, таблица, файл, деплой, настройка)
- Исследуешь тему, собираешь данные, анализируешь
- Работаешь с MCP-инструментами (БД, API, внешние сервисы)
- Вызываешь скиллы (слэш-команды)
- Пишешь, редактируешь или генерируешь код/контент
- Делаешь что-то что заняло больше 2 tool calls

**НЕ создавай ТОЛЬКО** для:
- Однострочный вопрос → однострочный ответ («какая сейчас дата?», «что значит X?»)
- Просмотр одного файла без действий

### Как вести журнал

Делай молча, не анонсируя.

1. Создай папку `project/ГГГГ-ММ-ДД-краткое-название/` в корне проекта.
2. Создай `log.md`:

```markdown
# Задача: [что делаем]
Дата: ГГГГ-ММ-ДД
Статус: в работе | завершено | заблокировано

## План
1. ...
2. ...

## Прогресс
### Шаг 1: [название]
- Что сделано: ...
- Результат проверки: ...
- Проблемы: ...

## Решения и находки
- [что сработало и почему — конкретно, с командами/параметрами]

## Обнаруженные навыки
- [если в процессе найден рабочий приём — описать здесь]
```

3. **Обновляй log.md по ходу работы** — после каждого значимого шага дописывай результат. Лог должен быть достаточно подробным, чтобы ДРУГАЯ сессия Claude могла прочитать его и полностью понять что было сделано, какие данные получены, и продолжить работу. Не просто «выполнено 15+ запросов» — а конкретные цифры, выводы, ключевые находки.
4. **Сохраняй навыки СРАЗУ как нашёл рабочее решение.** Не жди конца задачи.
   - Если ты потратил >2 попыток чтобы что-то заработало — это навык, запиши.
   - Если нашёл неочевидный параметр API, селектор, обходной путь — это навык, запиши.
   - Если делаешь что-то что потенциально понадобится повторно — это навык, запиши.
   - Запиши в раздел "Обнаруженные навыки" в log.md.
   - **СРАЗУ промоутируй в `~/.claude/skills/`** — НЕ ЖДИ конца сессии, НЕ СПРАШИВАЙ пользователя:
     1. Создай `~/.claude/skills/[категория]/SKILL.md` с полным содержимым навыка
     2. Добавь ссылку в MEMORY.md (секция "Навыки")
   - Формат SKILL.md:
     ```markdown
     # Навык: [название]
     ## Когда применять
     [в каких ситуациях]
     ## Как делать
     [пошаговая инструкция с конкретными командами, селекторами, параметрами]
     ## Почему это работает
     [объяснение — что не работает и почему именно так]
     ```
   - **НЕ промоутируй:** одноразовые селекторы для конкретного сайта, алиасы конкретных людей (это → память), факты без инструкции.

5. **Каждые 5 шагов — перечитай log.md** своей задачи. Защита от потери контекста при сжатии.

6. **Перед повторным решением задачи — ищи существующие навыки.**
   - Проверь `project/*/skill-*.md` — может это уже было решено.
   - Проверь `~/.claude/skills/` — там постоянные навыки.
   - Если нашёл — используй, не изобретай заново.

6.1. **Когда пользователь просит «вспомни», «найди», «продолжи», «мы делали X»:**
   - Сначала `ls project/` — прочитай log.md папок с подходящими названиями. Там план, прогресс, что осталось.
   - Если в логах не нашлось — ищи в JSONL сессиях (`~/.claude/projects/.../*.jsonl`): извлеки первые user-сообщения каждой сессии за нужный период.
   - Найдя задачу — прочитай log.md целиком, пойми где остановились, и продолжи с того места.

7. **При завершении задачи / сессии — обязательно:**
   - Обнови статус в log.md (`завершено` / `в работе` / `заблокировано`).
   - Допиши **итог** — что конкретно получилось, какие артефакты созданы (файлы, таблицы, ссылки).
   - Если задача не завершена — запиши **что осталось** и **с чего продолжать**.
   - Проверь что все навыки из log.md уже промоутированы в `~/.claude/skills/` (п.4). Если нет — промоутируй сейчас.

8. **Рефлексия — учись ПО ХОДУ работы, не только в конце.**

   **ТРИГГЕРЫ для немедленного сохранения в память (не жди конца сессии!):**

   a) **Пользователь поправил** («нет, не так», «я же сказал...», отклонил tool call):
      → СРАЗУ сохрани feedback-память. Что делал не так, почему, как правильно.

   b) **Пользователь одобрил неочевидное** («да, именно так», принял нестандартный подход без правок):
      → СРАЗУ сохрани feedback-память. Что сработало и почему это не очевидно.

   c) **Узнал новый факт о проекте** (новый партнёр, новый сервис, алиас, маппинг):
      → СРАЗУ сохрани project-память. Конкретные данные (имена, ID, маппинги) = ценность.

   d) **Заметил паттерн пользователя** (предпочитает таблицы, хочет HTML-отчёт, не хочет объяснений):
      → СРАЗУ сохрани user-память.

   **Как сохранять:** молча, без анонсов и вопросов. Создать файл в memory/, обновить MEMORY.md.
   **Когда НЕ сохранять:** очевидное, одноразовое, то что уже есть в memory или CLAUDE.md.
   **Проверка перед сохранением:** прочитай MEMORY.md — может это уже записано. Обнови существующий файл вместо создания дубля.

## 1. Верификация после каждого шага

После выполнения КАЖДОГО значимого действия — ПРОВЕРЬ результат перед переходом к следующему шагу:
- Файл создан? Прочитай его и убедись.
- Данные записаны? Запроси обратно и сверь.
- API вернул успех? Проверь что результат реально применился.
- Скрипт отработал? Проверь exit code И содержимое результата.

**Никогда не считай шаг выполненным без проверки.** "Команда не выдала ошибку" ≠ "всё сработало".

## 2. Целостность конвейера

При выполнении многоэтапных задач (A → B → C → D):
- Если шаг C сломался — СТОП. Не продолжай D и не повторяй A и B.
- Перед каждым новым шагом проверь что ПРЕДЫДУЩИЙ шаг реально завершён успешно.
- Каждые 10-15 итераций батч-операции — проверяй что весь конвейер всё ещё работает end-to-end.
- Если обнаружил что промежуточный шаг молча ломается — ОСТАНОВИ всю работу и сообщи пользователю.

## 3. Верность плану

Когда план согласован с пользователем:
- **Следуй плану шаг за шагом.** Не перескакивай, не меняй порядок без причины.
- **Если шаг не получается — ретрай тем же способом** (2-3 попытки), а не переключайся на другой подход.
- **Если после ретраев не работает — СПРОСИ пользователя.** Не изобретай альтернативный путь самостоятельно.
- **НИКОГДА не возвращайся к подходу, который уже был отвергнут** в текущей сессии.
- Если вынужден отклониться от плана — СНАЧАЛА сообщи пользователю что и почему, ПОТОМ действуй.
- **Сверяйся с log.md задачи** — там записано что уже пробовали и что сработало.

## 4. Контрольные точки в длинных задачах

При задачах с >10 действиями:
- Веди счётчик: "Шаг 5/12: загрузка изображений в хранилище"
- Каждые 5 шагов — краткий статус пользователю: что сделано, что впереди, есть ли проблемы.
- При батч-операциях (100+ элементов) — промежуточный отчёт каждые 20-30 элементов.
- **Каждые 5 шагов — перечитай log.md** чтобы не потерять контекст.

## 5. Обнаружение молчаливых сбоев

Красные флаги — ОСТАНОВИСЬ и проверь:
- Один и тот же вызов возвращает один и тот же результат 3+ раза подряд
- Размер результата резко изменился (было 50 элементов, стало 0)
- Операция "успешна" но при проверке данных нет
- Таймаут или пустой ответ от внешнего сервиса
- Повторяющиеся ошибки 4xx/5xx

При обнаружении: СТОП → записать проблему в log.md → диагностика → сообщить пользователю → ждать решения.

## 6. Язык общения

Всё на языке пользователя — общение, файлы, документы, журналы задач.
